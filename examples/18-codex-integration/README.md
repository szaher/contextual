# Example 18: Codex Integration

## What This Demonstrates

Three ways to use CtxKit with OpenAI Codex, ranging from full MCP
integration to a lightweight CLI fallback. Each option has different
tradeoffs in terms of features, setup complexity, and agent capabilities.

## Option A: MCP Registration

The recommended approach. Register the CtxKit MCP server with Codex so
it can call ctxl tools directly during a session.

### Setup

```bash
codex mcp add ctxkit -- ctxkit-mcp
```

This registers the `ctxkit-mcp` server with Codex. Codex spawns the
server when a session begins and can call any of the 10 available tools
(see [Example 16](../16-mcp-server) for the full tool list).

### Best-Practice Instruction Template

Add the following to your Codex instructions (system prompt or
AGENTS.md) so Codex knows how to use the ctxl tools:

```
## CtxKit Integration

Before starting any task:
1. Call ctxkit.context_pack with the task description and current
   working directory to get relevant project context.
2. Read the returned contracts carefully -- they are mandatory rules.
3. Check key_files entries for the files most relevant to the task.
4. Review gotchas for known sharp edges.

During the task:
5. Call ctxkit.log_event after each significant file edit or command
   execution.
6. If you need historical context, call ctxkit.memory.search with
   relevant keywords.

After completing the task:
7. Call ctxkit.propose_update with the list of changed files so that
   .ctx files can be kept current.
```

### What You Get

- Full access to all 10 ctxl tools
- Session tracking and event logging
- Memory search and storage
- Automatic `.ctx` update proposals
- Real-time context packs tailored to each request

## Option B: AGENTS.md Generation

A simpler approach that does not require MCP support. CtxKit generates
an `AGENTS.md` file containing a static snapshot of your project's
context. Codex reads `AGENTS.md` as part of its system prompt.

### Setup

```bash
ctxkit codex sync-agents
```

This command:
1. Scans all `.ctx` files in the repository
2. Builds a context pack using the default budget
3. Renders the context as Markdown
4. Writes it into `AGENTS.md` between marker comments

### The Marker Protocol

CtxKit uses markers to identify the section of `AGENTS.md` that it
manages. This allows you to keep your own content outside the markers:

```markdown
# AGENTS.md

Your custom instructions here. CtxKit will never touch this section.

<!-- CTXKIT:BEGIN -->
## Project Context (auto-generated by ctxkit)

### Summary
E-commerce platform monorepo with 3 packages...

### Contracts
- **code-quality** [REQUIRED]: All code must pass eslint with zero
  warnings, pass prettier format check...
- **auth-security** [REQUIRED]: All auth endpoints MUST validate
  JWT tokens...

### Key Files
- `package.json` -- Root package manifest with workspace scripts
- `src/auth/handler.ts` -- Main authentication request handler

### Decisions
- **ADR-001**: Monorepo with pnpm workspaces (accepted, 2026-01-05)
- **ADR-003**: JWT with refresh token rotation (accepted, 2026-01-08)

### Gotchas
- pnpm workspace dependencies must use 'workspace:*' protocol
- The auth middleware silently swallows errors from expired tokens

### Commands
- `pnpm -r build` -- Build all packages
- `vitest run` -- Run tests
- `eslint .` -- Lint
<!-- CTXKIT:END -->

More custom instructions below the markers. Also preserved.
```

### Idempotent Re-runs

Running `ctxkit codex sync-agents` multiple times is safe. Each run:
- Finds the existing `CTXKIT:BEGIN` / `CTXKIT:END` markers
- Replaces only the content between them
- Preserves everything before and after the markers
- Creates the markers if they do not exist yet (appended at the end)

```bash
# Safe to run repeatedly -- only the marked section changes
ctxkit codex sync-agents
ctxkit codex sync-agents
ctxkit codex sync-agents
```

### User Content Preservation

Content outside the markers is never modified:

```markdown
# AGENTS.md

## My Custom Rules          <-- preserved
Do not modify the database  <-- preserved
schema without approval.    <-- preserved

<!-- CTXKIT:BEGIN -->
(auto-generated content)    <-- replaced on each sync
<!-- CTXKIT:END -->

## My Workflow Notes        <-- preserved
Always run tests before     <-- preserved
submitting.                 <-- preserved
```

### Token Budget

The default budget for AGENTS.md generation is 8,000 tokens. This
controls how much context is included:

```bash
# Use the default budget (8000 tokens)
ctxkit codex sync-agents

# Override the budget
ctxkit codex sync-agents --budget 12000

# Minimal context
ctxkit codex sync-agents --budget 3000
```

Higher budgets include more entries (lower-scoring gotchas, additional
key files, more decisions). Lower budgets focus on contracts and
high-scoring entries only.

### Example AGENTS.md Output

```markdown
<!-- CTXKIT:BEGIN -->
## Project Context (auto-generated by ctxkit)

*Generated at 2026-03-01T10:00:00Z. Budget: 8000 tokens (5,420 used).*

### Summary
E-commerce platform monorepo with 3 packages:
- api: REST API (Hono + SQLite)
- shared: Types and validators
- web: React frontend (Vite)
Uses pnpm workspaces. TypeScript 5.x throughout.

### Contracts (REQUIRED -- always follow these)
- **code-quality** (scope: packages/**): All code must pass eslint
  with zero warnings, pass prettier format check, have TypeScript
  strict mode enabled, not use 'any' type without a comment.
- **type-compatibility** (scope: packages/shared/src/**): Breaking
  changes to shared types require updating BOTH api and web packages.
- **auth-security** (scope: packages/api/src/routes/auth*,
  packages/api/src/middleware/auth*): JWT tokens expire after
  15 minutes. Refresh tokens expire after 7 days and are single-use.
- **api-error-format** (scope: packages/api/src/routes/**): All API
  errors must return { error: { code, message } }.

### Key Files
- `package.json` -- Root package manifest with workspace scripts
- `pnpm-workspace.yaml` -- Workspace package definitions
- `packages/shared/src/types.ts` -- All shared TypeScript interfaces
- `packages/shared/src/validators.ts` -- Zod schemas for validation
- `packages/api/src/server.ts` -- Hono app setup and route mounting
- `packages/api/src/middleware/auth.ts` -- JWT validation middleware

### Decisions
- **ADR-001**: Monorepo with pnpm workspaces (accepted, 2026-01-05).
  Rejected: separate repos (duplication), npm workspaces (slower).
- **ADR-002**: SQLite for storage (accepted, 2026-01-05). Rejected:
  PostgreSQL (overkill for MVP).
- **ADR-003**: JWT with refresh token rotation (accepted, 2026-01-08).
  Rejected: session cookies (no mobile), JWT without refresh (risk).

### Gotchas
- pnpm workspace dependencies must use 'workspace:*' protocol, not
  version ranges.
- Build order matters: shared must build before api and web.
- The auth middleware must be registered BEFORE route handlers.
- better-sqlite3 operations are synchronous. Do not wrap in
  async/await.
- Zod schemas must use .strict() to reject unknown properties.

### Commands
- `pnpm -r build` -- Build all packages
- `pnpm -r --parallel dev` -- Start dev servers
- `vitest run` -- Run tests
- `eslint .` -- Lint
- `prettier --write .` -- Format

### Ignore (never read or reference these paths)
- .env, .env.*, *.pem, .aws/*
<!-- CTXKIT:END -->
```

## Option C: CLI Fallback

For environments where neither MCP registration nor AGENTS.md
generation is practical, you can use the ctxkit CLI directly with
JSON output. This is useful for custom scripts, CI pipelines, or
agents that support shell command execution.

### Get a Context Pack

```bash
ctxkit inject --json
```

Output (JSON):

```json
{
  "context_pack": {
    "total_tokens": 2340,
    "budget_tokens": 6000,
    "items": [
      {
        "source": "src/auth/.ctx",
        "section": "contracts",
        "name": "auth-security",
        "reason_code": "CONTRACT_REQUIRED",
        "tokens": 280,
        "content": "..."
      }
    ]
  }
}
```

### List Sessions

```bash
ctxkit sessions list --json
```

Output (JSON):

```json
{
  "sessions": [
    {
      "id": "sess_a1b2",
      "agent": "codex",
      "status": "completed",
      "requests": 3,
      "started": "2026-03-01T10:15:00Z",
      "ended": "2026-03-01T10:45:00Z"
    }
  ]
}
```

### Propose Updates

```bash
ctxkit propose --json
```

Output (JSON):

```json
{
  "proposals": [
    {
      "id": "prop_001",
      "file": "src/auth/.ctx",
      "section": "key_files",
      "type": "re-verification",
      "diff": "..."
    }
  ]
}
```

## When to Use Which Option

| Criteria | Option A: MCP | Option B: AGENTS.md | Option C: CLI |
|----------|---------------|---------------------|---------------|
| Setup complexity | Medium | Low | Low |
| Real-time context | Yes | No (static snapshot) | Yes (per-call) |
| Session tracking | Yes | No | Manual |
| Memory search | Yes | No | Yes |
| Automatic proposals | Yes | No | Manual |
| Works without daemon | No | Yes | Partial |
| Codex MCP support required | Yes | No | No |
| Best for | Full integration | Quick start, simple projects | Custom scripts, CI |

**Choose Option A** when you want the full ctxl experience with
real-time context packs, session tracking, and memory. Requires MCP
support in Codex and a running daemon.

**Choose Option B** when you want a quick, low-friction setup. The
static AGENTS.md approach works without a daemon and does not require
MCP support. Re-run `ctxkit codex sync-agents` periodically or in a
pre-commit hook to keep it current.

**Choose Option C** when you need programmatic access to ctxl from
scripts, CI pipelines, or agents that can run shell commands but do
not support MCP. The JSON output is stable and machine-readable.

## Combining Options

Options are not mutually exclusive. A common pattern:

1. Use **Option B** to generate a baseline AGENTS.md
2. Register the MCP server via **Option A** for real-time context
3. Use **Option C** in CI to validate `.ctx` files and check drift

```bash
# Initial setup
ctxkit codex sync-agents                    # Option B: baseline context
codex mcp add ctxkit -- ctxkit-mcp          # Option A: real-time tools

# CI pipeline
ctxkit validate --json                      # Option C: validate .ctx
ctxkit drift --json                         # Option C: check drift
```

## Best Practices

- **Start with Option B**: Generate an AGENTS.md first. This gives you
  immediate value with zero daemon dependency. Add MCP registration
  later when you need real-time features.

- **Keep AGENTS.md in version control**: Commit the generated file so
  all team members and CI get the same context.

- **Re-sync AGENTS.md in pre-commit hooks**: Add
  `ctxkit codex sync-agents` to your pre-commit hook so the file stays
  current with `.ctx` changes.

- **Use consistent budgets**: If you use both Option A and Option B,
  configure the same budget for both so the context is consistent.

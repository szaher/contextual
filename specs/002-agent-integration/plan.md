# Implementation Plan: CtxKit Agent Integration

**Branch**: `002-agent-integration` | **Date**: 2026-03-01 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/002-agent-integration/spec.md`

## Summary

Build two first-class agent adapters (Claude Code plugin + Codex adapter) and a shared MCP server so developers can run CtxKit from inside Claude Code and Codex with automatic context injection and traceability. The MCP server wraps the existing daemon API as stdio-based JSON-RPC tools. The Claude Code plugin uses hooks for automatic lifecycle integration. The Codex adapter uses MCP tools plus generated `AGENTS.md` files for zero-friction memory injection.

## Technical Context

**Language/Version**: TypeScript 5.x / Node.js 20+
**Primary Dependencies**: `@modelcontextprotocol/sdk` ^1.27 (MCP server), `zod` ^3.25 (schema validation), `@ctxl/core` (workspace dependency), `commander` ^13 (CLI extensions)
**Storage**: Existing better-sqlite3 via `@ctxl/daemon` (no new storage layer)
**Testing**: Vitest (integration + E2E), real filesystem temp dirs, spawned daemon processes
**Target Platform**: macOS / Linux (local developer machines)
**Project Type**: Monorepo packages (library + CLI extension + plugin)
**Performance Goals**: Hook response time <500ms (SC-002), graceful degradation within 2s (SC-006), skill response <3s (SC-009)
**Constraints**: Local-first (no network), stdio transport only for v1, must not block agent workflow on failure
**Scale/Scope**: Single developer workstation, any repo size supported by Feature 001

## Constitution Check

*GATE: Must pass before Phase 0 research. Re-check after Phase 1 design.*

### I. Local-First, Private-by-Default — PASS

- MCP server uses stdio transport (no network).
- All data flows through the local daemon. No external API calls.
- Hook scripts run locally on the developer machine.
- Generated `AGENTS.md` files are repo-local and committed to version control.

### II. Repository Truth Over Guessing — PASS

- MCP server wraps existing daemon API routes (sessions, context-pack, proposals, drift, audit). No new data access patterns invented.
- Hook scripts call existing `@ctxl/core` functions via the daemon HTTP API.
- `AGENTS.md` generation reads existing `.ctx` files — no invented content.
- Reuses existing `buildContextPack`, `loadProfile`, `detectSecrets`, `redactSecrets` from `@ctxl/core`.

### III. Transparent, Inspectable Context Injection — PASS

- Every injected context item includes source `.ctx` path, reason codes, and pack ID (FR-028).
- Omitted items list is always returned by context pack tools.
- Hook `additionalContext` includes compact headers with pack ID and token estimate.
- Audit trail via PostToolUse/PostToolUseFailure logging.

### IV. Deterministic, Budgeted Context — PASS

- Context packs are generated by the same `buildContextPack` function used by Feature 001.
- Token budgets are configurable per hook (FR-021a) and per MCP tool call.
- Same inputs + same repo state = same pack (determinism inherited from core).

### Quality & Testing Standards — PASS

- E2E tests: spawn daemon + MCP server, send JSON-RPC messages, verify responses.
- Integration tests: hook scripts with mock stdin/stdout, daemon API calls.
- Unit tests targeted at: MCP message parsing, hook JSON I/O serialization, AGENTS.md generation/merge logic.
- Test realism: real filesystem temp dirs, real sqlite, spawned processes.

### Context & Memory Standards — PASS

- Secret redaction applied to all generated `AGENTS.md` files (FR-024, FR-029).
- `.ctx` updates proposed as diffs, never silently rewritten.
- Ignore policies respected in all context pack generation paths.

### Operational Safety — PASS

- All hooks have configurable timeouts (FR-020, default 5s).
- Hooks fail gracefully — empty context returned on error (FR-031).
- MCP server returns structured error objects on daemon unavailability (FR-010).
- SessionEnd hook properly closes sessions with atomic writes.
